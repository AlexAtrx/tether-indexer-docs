# Access the rumble-dev server

This skill captures the exact access pattern and conventions for the Rumble
dev VM. It is self-sufficient: no external setup is required, every command
below works from a fresh session.

## Identity model (read before doing anything)

- Local SSH alias: **`rumble-dev`** (no entry in `~/.ssh/config` — resolves
  via Tailscale DNS `rumble-dev.tail8a2a3f.ts.net` → `100.99.150.14`).
- Login user: **`alexa`** (matches the local username; default SSH key is
  picked automatically). Alex is in `google-sudoers`, so `sudo` is
  passwordless.
- Service user: **`work`**. Owns `/home/work/wdk/` (the cloned tetherto
  repos) and `/home/work/.pm2/` (PM2 daemon, logs, dump file). All
  wdk-be-deploy operations and PM2 process commands MUST run as `work`,
  otherwise you'll be talking to a different (empty) PM2 instance running
  under `alexa`.

To verify connectivity once before a session:

```bash
ssh rumble-dev 'whoami; hostname; uptime'
# expect: alexa / rumble-dev / load average ...
```

## How to run anything as `work`

Two patterns. Use the second one for any multi-line script.

### Pattern A — one-liner

```bash
ssh rumble-dev 'sudo -u work pm2 jlist | jq -r ".[] | .name" | sort'
```

Single quotes around the remote command keep `$var`/`$()` inert locally and
let the remote bash do the expansion. Fine for commands without nested
quoting.

### Pattern B — multi-line script (RECOMMENDED)

```bash
ssh rumble-dev 'sudo -u work bash -s' <<'REMOTE'
set -u
for d in /home/work/wdk/wdk-indexer-wrk-btc /home/work/wdk/rumble-ork-wrk; do
  echo "==== $d ===="
  ls "$d/worker.js" 2>/dev/null
done
REMOTE
```

The **single-quoted heredoc tag (`<<'REMOTE'`) is required**. Without it,
your local shell expands `$d` (to empty) before the script ever reaches the
server. Pattern A with nested `bash -lc "..."` runs into the same trap
because the outer double quotes still interpolate `$`. Heredoc with quoted
EOF tag dodges both problems.

Inside the heredoc, write normal bash. The script lives only in the
remote shell's stdin — nothing is written to disk on the server.

### When NOT to use `sudo -u work`

PM2 only matters for the `work` user. Things that are global to the box
(disk usage, kernel logs, system load, network sockets) can run as `alexa`
without `sudo -u work`:

```bash
ssh rumble-dev 'df -h /; free -m; uptime'
ssh rumble-dev 'ss -tlnp 2>/dev/null | head'   # no sudo needed for own sockets
```

## Layout on the server

Owner is `work` for everything below.

```
/home/work/
├─ .wdk-be-conf.json              # generated by `wdk-be-deploy init`, holds rpc keys
├─ .hyper-hosts                   # hyper RPC host map (rpc keys for quick CLI access)
├─ .pm2/
│  ├─ pm2.log                     # daemon log: Stopping/exited/online events
│  ├─ pm2.pid
│  ├─ dump.pm2                    # process list snapshot for `pm2 resurrect`
│  └─ logs/                       # per-process out/error logs, rotated daily
└─ wdk/                           # all cloned tetherto repos
   ├─ wdk-devops/wdk-be-deploy/   # CLI used to spawn workers (npm-linked as `wdk-be-deploy`)
   ├─ wdk-indexer-wrk-btc/        # one indexer repo per chain family
   ├─ wdk-indexer-wrk-evm/
   ├─ wdk-indexer-wrk-ton/
   ├─ wdk-indexer-wrk-tron/
   ├─ wdk-indexer-wrk-spark/
   ├─ wdk-indexer-processor-wrk/
   ├─ rumble-data-shard-wrk/      # rumble overlay of the shard
   ├─ rumble-ork-wrk/
   ├─ rumble-app-node/            # the only HTTP entrypoint for wallet/user API
   ├─ rumble-promo-wrk/
   └─ docker/                     # local mongo/redis compose for dev (if used)
```

Process naming convention (today on rumble-dev — may change as topology
evolves):

- `idx-<chain>-<token>-proc-w-0` and `idx-...-api-w-0-0` — indexer proc/api
- `processor-<chain>-<token>-w-0-<N>` — Redis-stream processor
- `shard-proc-w-<rack>` and `shard-api-w-<rack>-0` — data shard
- `ork-w-<rack>` — ork (Hyperswarm router / lookup)
- `app-<port>` — HTTP app node (3000 / 3001)
- `promo-proc-w-0` / `promo-api-w-0-0` — promo worker

If you need to filter by family, use a regex: e.g. `select(.name |
test("^(idx-|processor-|shard-|ork-w-|app-|promo-)"))`. Do NOT rely on
`startswith` — the deploy script has historically broken when names
changed (the Feb 2026 `startswith("ork-w-")` filter returned 0 ids when
orks were called `ork-0/1/2`).

## Common recipes

### List PM2 processes

```bash
ssh rumble-dev 'sudo -u work pm2 list'
ssh rumble-dev 'sudo -u work pm2 jlist | jq -r ".[] | [.pm_id, .name, .pm2_env.status, .pm2_env.restart_time] | @tsv"'
```

### Inspect kill_timeout / listen_timeout per process

```bash
ssh rumble-dev 'sudo -u work pm2 jlist | jq -r ".[] | [.pm_id, .name, .pm2_env.kill_timeout, .pm2_env.listen_timeout] | @tsv"'
```

(Background: every wdk process is started with `--kill-timeout 300000`
from `wdk-devops/wdk-be-deploy/lib/common.flags.js`. Useful to confirm
when investigating slow restarts.)

### Tail logs

```bash
# Live tail of one process (stays open — Ctrl-C to end)
ssh rumble-dev 'sudo -u work pm2 logs idx-bitcoin-proc-w-0 --lines 100'

# Snapshot tail of out + error
ssh rumble-dev 'sudo -u work tail -n 200 /home/work/.pm2/logs/ork-w-0-out.log /home/work/.pm2/logs/ork-w-0-error.log'
```

PM2 rotates logs daily (`pm2-logrotate` module), so historical logs are at
`/home/work/.pm2/logs/<name>-out__YYYY-MM-DD_HH-MM-SS.log`.

### Read code

Read from `/home/work/wdk/<repo>/...` like any local file. Worker source
lives under `workers/`, library code under `lib/` or `workers/lib/`,
config under `config/<env>.<workerName>.json` (env is usually
`production`). The boot wrapper is the package
`@bitfinex/bfx-svc-boot-js`; the base class is `@bitfinex/bfx-wrk-base`.

When you need a different ref (branch, tag, PR) or the repo isn't on the
box, fall through to the `read-remote-repo` skill — don't try to `git
fetch` on the server.

### Restart processes (DISRUPTIVE — confirm first)

```bash
# Single process (~1 s for fast services, ~7 s for orks)
ssh rumble-dev 'sudo -u work pm2 restart ork-w-1'

# Whole wdk fleet, sequentially via xargs (this is the same path the
# deploy script uses; on today's trimmed topology it takes ~50 s)
ssh rumble-dev 'sudo -u work bash -s' <<'REMOTE'
pm2 jlist | jq -r '.[] | select(.name | test("^(idx-|processor-|shard-|ork-w-|app-|promo-)")) | .pm_id' | xargs pm2 restart --silent
REMOTE
```

Always confirm with Alex before restarting prod services. Dev is normally
safe to restart, but coordinate if a PR is mid-smoke-test.

### Run `wdk-be-deploy` commands (also disruptive)

`wdk-be-deploy` is npm-linked, so it's on `work`'s PATH:

```bash
ssh rumble-dev 'sudo -u work bash -lc "cd /home/work/wdk && wdk-be-deploy --help"'
```

Use the same Pattern B heredoc form for anything multi-line. The CLI's
own README at `/home/work/wdk/wdk-devops/wdk-be-deploy/README.md` is the
canonical reference for what each subcommand does (indexer / processor /
shard / ork / app / compliance / rotate / hyperhost / version).

### Check service health quickly

```bash
ssh rumble-dev 'sudo -u work bash -s' <<'REMOTE'
echo "--- non-online processes ---"
pm2 jlist | jq -r '.[] | select(.pm2_env.status != "online") | [.pm_id, .name, .pm2_env.status] | @tsv'
echo "--- error logs with content (size and last line) ---"
for f in /home/work/.pm2/logs/*-error.log; do
  n=$(wc -l < "$f"); [ "$n" -gt 0 ] && echo "$f  ($n lines)  $(tail -1 "$f" 2>/dev/null | head -c 200)"
done
REMOTE
```

## Cleanup rules (HARD REQUIREMENT)

Alex's standing instruction: **never leave AI-authored files, scripts, or
markers on the server.** Concretely:

1. **Prefer stdin / heredoc over writing scripts to disk.** Pattern B
   above runs your script via `bash -s` from stdin — no file ever lands on
   the box.
2. If you absolutely must write a temp file (e.g. a long script that
   doesn't fit a single ssh call, or something that needs to survive an
   ssh disconnect), put it under `/tmp/` only, use a name without any
   `claude` / `ai` / `bot` markers (`/tmp/inv-<short>.sh` is fine), and
   `rm -f` it before the session ends. Confirm removal with `ls /tmp/...`
   returning nothing.
3. Never write to `/home/work/`, `/etc/`, `/opt/`, or any path owned by
   `work`/`root` outside `/tmp/`. Even read-only investigations should
   not create files in repo working trees.
4. Do not commit on the server. The clones under `/home/work/wdk/` are
   deployment checkouts. Any code change goes through normal PR flow on
   your laptop, then a pull on the server.
5. Avoid `--silent` redaction games on outputs you save locally — if
   tokens / rpc keys / capability strings show up in command output,
   redact them when writing to local docs.

Quick self-check before ending a session that touched the server:

```bash
ssh rumble-dev 'ls -la /tmp/ 2>/dev/null | grep -vE "^(d|total|.*\.lock$|.*-1[0-9]{3}/$|.*\.X|\.font|systemd|claude|ssh)" | head'
```

(Adjust the grep; the goal is to spot files newer than the session that
have AI-ish names.)

## Reproducing the slow-restart investigation (WDK-1229)

For reference, the WDK-1229 root cause was found by:

1. `pm2 jlist | jq` to confirm `kill_timeout=300000` on every wdk process.
2. Reading `wdk-devops/wdk-be-deploy/lib/common.flags.js` to find where
   that timeout is set.
3. Reading `node_modules/@bitfinex/bfx-svc-boot-js/index.js` (under one of
   the worker repos) to find the SIGINT handler's early-return bug when
   `hnd.active === 0`.
4. Reading `node_modules/@bitfinex/bfx-wrk-base/base.js` to find the
   `lockProcessing` poll with no timeout in `stop()`.
5. `~/.pm2/pm2.log` for actual Stop→exit timings, then
   `pm2 restart <name>` to measure live wall time, then a full sequential
   restart via the same `xargs pm2 restart` path the deploy script uses.

Full writeup:
`_tether-indexer-docs/_tasks/24-20-may-26-WDK-1229-rumble-dev-address-environment-slow-restart/ROOT-CAUSE.md`.

## Gotchas

- `bash -lc "..."` nested in an outer double-quoted ssh string lets
  `$vars` leak to the local shell. Always use the heredoc pattern with a
  quoted `<<'REMOTE'` tag for multi-line work.
- `pm2` commands run as `alexa` see a different (usually empty) PM2
  daemon. Always `sudo -u work`.
- `pm2 restart <id>` returns when the new process is forked (`online`),
  not when the worker has finished booting. For real boot time, watch the
  out log for the worker's own "ready" message (e.g. `"Ork ready"`,
  `"Worker started as master"`, indexer's first `"started processing
  blocks"` line).
- The Tailscale DNS name `rumble-dev.tail8a2a3f.ts.net` only resolves
  when the local Tailscale client is up. If `ssh rumble-dev` errors with
  "Could not resolve hostname", check `tailscale status`.
- `bc` is not installed on the box. Use `$((t1-t0))` or `awk` for arithmetic.
