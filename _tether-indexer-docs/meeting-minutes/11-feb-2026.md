## Technical Discussion

- Backend stores only a local hash/fingerprint of the seed for grouping; no raw seed or derived key persisted.
- Balance endpoint retries provider up to three times; on partial provider failure returns null for affected token/chain.
- Null values trigger a fallback fetch from the RPC proxy endpoint to fill missing balances.
- api/v1/wallet/balances aggregates balances for all wallets, blockchains, and tokens in a single fast call.
- Token‑transfer endpoint supports queries by wallet ID or by user ID; seed‑phrase login currently lacks a user‑ID path.
- Proposed solution: deterministically derive a seed‑level identifier from the seed derivation path and aggregate history across all wallets linked to that seed.
- Current transaction merging streams sort by timestamp; only one blockchain string is accepted per request—multiple chains need separate calls.

# v1‑v3 Migration with Backend Readiness Gate

## Action Items

- Add a readiness gate so shards reply only after Autobase discovery and Mongo sync completes
- Build a consistency test suite that validates wallet and channel counts across shards during startup
- Backfill v1 wallet derivation indexes into Mongo during shard startup, using Autobase metadata
- Monitor Spark RPC latency and Redis cache hit rates after re‑enabling full parallel balance fetches
- Run v1 → v3 migration tests on Noxton staging (create v1 account, fund tip jars, upgrade via seed phrase, verify balances, tip jars, transaction history)
- Align on a single backend user‑id field for trace‑id across all endpoints; update mobile team on the change
- Share the updated SPEC.md (Transaction History API v2) with the team and confirm no further spec changes needed

## Key Discussion Points

- Spark balance reset issue originates in the balance‑fetch pipeline, not the transaction‑history aggregation layer
- Increased Spark rate limit now allows batch balance fetching; need to watch for latency and cache performance
- Transaction History API v2 spec defines read‑side aggregation and on‑chain grouping but doesn’t fix live balance bugs
- Notification token format bug resolved; FCM token now sent on both iOS and Android after full login
- Migration from HyperDB to Mongo introduced rare cache‑poisoning edge case causing “wallet not found” errors
- BTC address rollback needed first to ensure address consistency before v1 → v3 migration testing
- Current API sometimes returns internal error codes instead of generic “internal server error” – needs standardization

## Decisions Made

- Implement the readiness gate as a small change to guarantee consistent shard responses during startup
- Use client‑provided index hint, verified against Autobase metadata, before persisting to Mongo (safer than guessing)
- Proceed with v1 → v3 migration testing on staging using fresh v1 accounts (avoid legacy accounts)
- Merge BTC rollback into staging first, then create new v1 accounts with tip jars for migration validation
- Standardize on the backend user‑id field for trace‑id across all services and mobile SDKs

## Open Issues / Risks

- Spark balance still occasionally resets to zero before returning to correct value – need deeper RPC diagnostics
- Notification flow depends on correct token format; monitor for any regression after next release
- Arbitrum provider configuration still unclear – confirm whether we use public API or premium account
- Need comprehensive integration tests for all workers to catch multi‑request edge cases that caused prior inconsistencies
- Potential duplicated tip jars if backend incorrectly assumes wallet existence during migration – guard with index verification
