## Max Button & Fee Calculation

- “Max” should show the full wallet balance, but the actual send amount must account for fees + a safety buffer.
- Prefetching fees on the input screen is okay if inputs are debounced and the last estimate is cached to avoid race conditions.
- Static minimum send threshold (e.g., disallow sending &lt;$1) simplifies logic and removes dynamic fee recalculations.
- Buffer currently set to \~20 % of estimated fee; may be stale but prevents under‑funded transactions.
- UX suggestion: display max amount, then on the confirmation screen show the net amount after fee deduction.

## Swap & Send Flow Issues

- Swap screen “Max” leads to unstable behavior: amount changes after fee recalculation, causing mismatched input vs. confirmation.
- Send screen shows static balance; no pre‑fetch of fees, so “Max” remains consistent.
- Users see a mismatch between the amount shown on the final transaction and the amount they entered.
- Proposed approach: keep fee calculation on the confirmation screen (as other apps do) to avoid UI flicker and race conditions.
