# Bug: `GET /token-transfers` type filter doesn't work

**Ticket**: WDK-1287 | Priority: High | Assignee: Alex Atrash
**Asana**: https://app.asana.com/1/45238840754660/project/1210540875949204/task/1213746973227038

---

## Bug Summary

Endpoint `GET /api/v1/users/:userId/token-transfers?type=received` returns **both** "sent" and "received" entries for the same on-chain transaction when the transfer occurs between two wallets owned by the same user. The `type` query parameter filter is not working.

## Reproduction

```
GET /api/v1/users/PEGuyaDBPDo/token-transfers?walletTypes=channel&walletTypes=user&type=received&sort=desc&limit=50
```

Returns tx `0x09eb6f...` twice: once with `"type": "sent"` (walletId `412f7683-...`) and once with `"type": "received"` (walletId `83ed8531-...`).

---

## Root Cause

**File**: `wdk-data-shard-wrk/workers/api.shard.data.wrk.js`
**Method**: `getUserTransfers` (line 312)

### Two bugs combine:

### Bug 1: Matcher filters on global address list instead of computed `type` (lines 360-367)

```js
const matcher = (tx) => {
  // ...
  if (type && type.toLowerCase() === 'sent' && !walletAddresses.includes(tx.from)) return false
  if (type && type.toLowerCase() === 'received' && !walletAddresses.includes(tx.to)) return false
  return true
}
```

The matcher checks whether `tx.from` / `tx.to` is in `walletAddresses` -- the **combined** addresses of **all** the user's wallets. For a self-transfer (Wallet-1 -> Wallet-2, both owned by user):

- `tx.from` = Wallet-1 address (in `walletAddresses`) -> "sent" check passes
- `tx.to` = Wallet-2 address (in `walletAddresses`) -> "received" check passes

So with `?type=received`, the matcher checks `walletAddresses.includes(tx.to)` -- which is `true` for **both** the Wallet-1 iterator record AND the Wallet-2 iterator record, because `tx.to` is the same in both. Both pass the filter.

The matcher should instead filter on the **already-computed** `type` field (set per-wallet on line 380), not re-derive it from the global address list.

### Bug 2: No deduplication across wallet iterators (lines 404-429)

The method creates one iterator per wallet (line 399) and merge-sorts them. When a transfer goes from Wallet-1 to Wallet-2 (same user), two DB records exist (one per wallet). The merge loop emits **both** without deduplication by `transactionHash + transferIndex`.

---

## Proposed Fix

### Fix 1: Filter on computed `type` field (primary fix)

```js
// BEFORE (lines 360-367):
const matcher = (tx) => {
  if (!tx || typeof tx !== 'object') return false
  if (token && tx.token !== token) return false
  if (blockchain && tx.blockchain !== blockchain) return false
  if (type && type.toLowerCase() === 'sent' && !walletAddresses.includes(tx.from)) return false
  if (type && type.toLowerCase() === 'received' && !walletAddresses.includes(tx.to)) return false
  return true
}

// AFTER:
const matcher = (tx) => {
  if (!tx || typeof tx !== 'object') return false
  if (token && tx.token !== token) return false
  if (blockchain && tx.blockchain !== blockchain) return false
  if (type && tx.type !== type.toLowerCase()) return false
  return true
}
```

This works because `tx.type` is already computed per-wallet on line 380 **before** the matcher runs on line 384:
```js
const txWithCalculatedFields = {
  type: walletIdToAddresses[wallet.id].includes(tx.from) ? 'sent' : 'received',
  ...tx
}
if (matcher(txWithCalculatedFields)) { ... }
```

### Fix 2: Deduplicate across wallets (secondary fix)

Add dedup in the merge loop to prevent the same on-chain transfer from appearing twice when both sender and receiver wallets belong to the same user:

```js
// Add before the merge loop (after line 402):
const seen = new Set()

// Modify the merge loop body (lines 420-424):
if (skipped < skip) {
  skipped++
} else {
  const dedupeKey = `${nextTransfer.transactionHash}:${nextTransfer.transferIndex ?? 0}`
  if (!seen.has(dedupeKey)) {
    seen.add(dedupeKey)
    transfers.push(nextTransfer)
  }
}
```

**Note on dedup**: Whether to deduplicate depends on product intent. If the API should show both the "sent" and "received" side of a self-transfer (two wallet perspectives), skip Fix 2. If it should show each on-chain transfer only once, apply Fix 2. Fix 1 is required regardless.

---

## Impact

- Fix 1 is a 2-line change, zero risk of regression -- the matcher simply uses the field that's already computed
- Fix 2 is optional depending on desired behavior for self-transfers
- No schema or migration changes needed
- No changes to other services (ork, indexer, app-node)
