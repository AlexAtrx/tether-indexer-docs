# Root Cause

- BTC indexing is output-based. `wdk-indexer-wrk-btc/workers/lib/providers/rpc.provider.js:124-193` emits one transfer row per `vout`, so a single spend produces both:
  - payment output
  - change output
- Shard sync persists every returned row with no BTC-specific filtering in `wdk-data-shard-wrk/workers/proc.shard.data.wrk.js:895-943`.
- The user/wallet transfer endpoints classify direction only from wallet ownership of `from`:
  - `wdk-data-shard-wrk/workers/api.shard.data.wrk.js:364-365`
  - `wdk-data-shard-wrk/workers/api.shard.data.wrk.js:379-380`
  - `wdk-data-shard-wrk/workers/api.shard.data.wrk.js:484-485`
  - `wdk-data-shard-wrk/workers/api.shard.data.wrk.js:498-499`
- That logic is wrong for BTC change outputs. If `from` and `to` are both addresses of the same wallet, the row is still labeled `sent`, so one logical BTC send becomes multiple sender-side rows and an inflated apparent sent amount.
- Receiver-side looks correct because only the payment output matches the receiver wallet.

# Proposed Fix

- Fix the legacy classification in `wdk-data-shard-wrk/workers/api.shard.data.wrk.js` with wallet-aware logic:
  - `change`: `from` and `to` are both addresses in the same wallet
  - `sent`: `from` is wallet-owned and `to` is external
  - `received`: `to` is wallet-owned and `from` is external
- For backward compatibility, filter `change` rows out of legacy `/token-transfers` responses instead of exposing a new public `type=change` by default.
  - Current route schema in `wdk-app-node/workers/lib/server.js:559` and `wdk-app-node/workers/lib/server.js:605` only allows `sent|received`.
- Add regression coverage for both wallet and user endpoints using a BTC payment output plus a same-wallet change output.

# Important Limitation

- If the ticket expects the legacy endpoint to also return the BTC network fee, the current shard storage is not enough for a reliable fix:
  - shard HyperDB wallet-transfer schema only stores flat transfer rows in `wdk-data-shard-wrk/workers/lib/db/hyperdb/build.js:65-83`
  - fee/input/change context is not modeled there
- So:
  - classification/filtering fixes the inflated sender amount
  - accurate fee display still needs extra BTC context persisted end-to-end or a newer processed-history path
