- `_broadcastPayoutTx`
  - Keep durability invariant: sign -> persist hash -> broadcast, which is explicitly documented and should not be broken 
  - Verify all downstream callers respect the fact that a row can be `paying` with or without a persisted hash depending on where execution interrupted.
  - Remove redundant passing of `txHash` into later functions if the hash is already committed on the row and the callee can identify the row by `payoutId`.
  - Be explicit in code/comments that `broadcast` means provider submission, not merely local signing.

- Naming around mined / landed / reverted
  - Rename misleading `landed` / `not landed` naming.
  - `receipt.status === 1` should map to something like `mined_successfully` or `landed_successfully`.
  - `receipt.status === 0` should map to `mined_but_reverted` or `landed_but_reverted`, because the tx did land on-chain but failed semantically.
  - This matches the documented resolver semantics where `payout + status=1` becomes mined plus settled callback, and `payout + status=0` becomes failed plus failed callback  

- `_observeBroadcastedV2Payouts`
  - Keep conceptual separation: this is the reconciliation loop for post-broadcast lifecycle, and it is expected to be load-bearing for the happy path after `broadcast`  
  - Reviewer feedback: naming `stale` is overloaded here. In observer, `stale` really means “old enough to reconcile again,” not necessarily “broken.”
  - Consider renaming the selector or comments to reflect “observation eligibility threshold” instead of failure semantics.
  - Revisit the observer stale threshold value from a UX perspective. Current docs show `payoutStaleThresholdMs` defaulting to 5 minutes, which may be too conservative for fast chains and delays callbacks  
  - Consider chain-specific thresholds or a shorter observer interval rather than removing the guard entirely.
  - Preserve rule that `broadcast` rows can legitimately exist before `mined`; do not conflate those states.

- `_recoverStuckV2Payouts`
  - Revisit whether `payoutStaleThresholdMs` really makes sense for interrupted `paying` work.
  - Reviewer’s point was strong: this path is not waiting on chain finality, it is recovering a local interrupted execution. Docs currently define it as scanning stale `paying` rows older than threshold 
  - TODO: review if the threshold should be significantly smaller or replaced with a more explicit freshness/race guard.
  - If kept, document clearly that the threshold exists only to avoid racing fresh in-flight work or cross-process overlap.
  - Validate the branch semantics:
    - final receipt -> resolve
    - provider still knows tx -> heal back to `broadcast`
    - retries exhausted -> cancel
    - otherwise resubmit with bumped fees 
  - Confirm no old callback-enqueue behavior still exists in the “provider knows tx” path. Docs explicitly say it should only heal back to `broadcast`; observer owns final settlement 

- `_classifyHashes`
  - Keep this as the single primitive for receipt scan across all persisted hashes. Docs explicitly define it as returning the first conclusive receipt across the row’s hashes, tagged `payout` or `cancel` by `receipt.to` 
  - Add explicit tests for multiple hashes on one row, including payout attempts plus cancel attempts.
  - Add explicit test for “earlier payout mined, later cancel attempted” to ensure on-chain truth resolves as payout, not cancel.
  - Add explicit test for “cancel mined successfully” to ensure it resolves as failure, not settlement.
  - Consider making ordering policy more explicit if multiple hashes can be conclusive on-chain, so reconciliation behavior is deterministic and reviewable.
  - Keep `receipt.to` classification as the single source of truth for payout-vs-cancel; docs call this the keystone of cancel safety 

- `_resolveClassifiedReceipt`

  - Improve naming and warning semantics so reviewers understand this can still represent happy business outcome even though it is reached via reconciliation flow.
  - Clarify in code/comments that warn-level visibility is about reconciliation path visibility, not necessarily application error.
  - Ensure `phase` tagging remains useful and consistent for ops visibility across recovery, observer, and cancel-exhaust flows 

- `_cancelExhaustedPayout`

  - Review whether any current implementation path is incorrectly depending on `lastHash` instead of all persisted hashes.

- `_findHighestFeeKnownTx`

- `_finalizeCancellingV2Payouts`
  - Major refactor target.
  - Remove duplicate receipt-scanning and outcome-resolution logic. This function should ideally reuse `_classifyHashes` and `_resolveClassifiedReceipt` rather than partially re-implementing them.
  - The intended policy is narrow and should stay narrow:
    - payout tx mined -> on-chain truth wins -> settled
    - payout tx mined with revert -> failed
    - cancel tx mined -> failed
    - cancel dropped -> rebroadcast cancel with bumped fees
    - past `cancelMaxRetries` -> alert and leave in `cancelling` for ops review 
  - Review redundant `getTransaction` logic:
    - if all hashes were already scanned for known/pending state, avoid a second `getTransaction(lastHash)` unless it adds unique information.
  - Fix potential design mismatch where logic seems to rely on last transaction hash instead of all persisted hashes.
  - Reviewer concern to validate: current pending-cancel path may stall indefinitely if cancel tx remains known and pending, because rebroadcast only happens after provider stops knowing it.
  - Add explicit `cancelPendingMaxMs` style policy to avoid relying purely on mempool/provider eviction.
  - Consider same-nonce cancel fee-bump while previous cancel tx is still known, instead of waiting for provider drop.
  - Review V1 for precedent. Reviewer said earlier implementation likely retried stuck nonce replacement multiple times until success.
  - Add tests for:
    - payout mined late while cancelling
    - cancel mined
    - cancel dropped and rebroadcasted
    - cancel remains pending too long
    - repeated cancel fee bumps
    - retries exhausted with nonce still blocked
  - Confirm this function cannot silently stall forever in `cancelling`.
