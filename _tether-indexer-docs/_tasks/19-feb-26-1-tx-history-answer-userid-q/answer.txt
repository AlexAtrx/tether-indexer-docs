Solution: Processor includes { address: userId } map in grouped message

Dev 1's proposal is correct. The processor already resolves address -> walletId -> shardId for routing.
The lookup document also carries userId but we currently discard it.
Include it in the message so the shard can populate fromUserId/toUserId without breaking shard isolation.

Changes needed (stream path)

1. wdk-indexer-processor-wrk (indexer.processor.wrk.js ~line 287)
   - For each unique from/to address in the tx group, resolve address -> walletId (already done).
   - Resolve walletId -> userId from the same lookup entry (userId field already present in wdk_ork_lookups doc, currently ignored).
   - Attach map to message:
     { txHash, transfers, walletId, addressUserIds: { "<address>": "<userId>", ... } }
   - Unknown/external addresses stay absent from the map.

2. wdk-data-shard-wrk
   - proc.shard.data.wrk.js (~line 1278): extract addressUserIds from parsed message, pass to transfer processor.
   - transfer.processor.js (~line 32-33): replace hardcoded nulls:
     fromUserId = addressUserIds[from] || null
     toUserId = addressUserIds[to] || null
   - Tests: update transfer.processor.test.js assertions (lines 403-404) to validate populated userIds.

3. No changes to ork, app-node, or API response shape — fields already exist as null.

Gap: polling path

Dev 2 flagged: "only issue is in polling logic, in this case shard is calling indexer directly — on that we won't have userIds."

The shard's syncWalletTransfersJob (proc.shard.data.wrk.js ~line 804) calls blockchainSvc.getTransfersForWalletsBatch() directly, bypassing the processor. This path has no access to the ork lookup tables.

Two sub-questions to clarify before solving:

A) Does the polling path feed the new transfer-history API?
   - Currently polling stores to walletTransferRepository (wallet_transfers_v2 collection).
   - The new /transfer-history endpoints read from walletTransfersProcessedRepository (wallet_transfers_processed collection).
   - If these are separate collections and polling only feeds the legacy endpoint, then polling doesn't affect the new API at all — the gap only matters for the legacy /token-transfers path.

B) If polling DOES need to feed processed records (backfill/catch-all scenario):
   Options:
   1. Shard resolves owner userId from wallet context (it already has the wallet object).
      Counterparty userId stays null for polled transfers — acceptable degradation since polling is the fallback path, not the primary one.
   2. Route polled transfers through the processor instead of storing them directly.
      Shard publishes to a stream, processor enriches (including addressUserIds), sends back via the normal grouped path.
      Cleaner but requires plumbing changes.

Recommendation: start with the stream path (Option C). For polling, option B1 (owner from wallet, counterparty null) is the pragmatic choice unless polling is expected to be the primary path for any use case.

Why not the shared lookup lib approach

Dev 2's original suggestion (shared readonly lookup lib reused in both processor and shard) would require the shard to connect to the ork-level MongoDB (wdk_ork_lookups, wdk_ork_wallet_id_lookups). This crosses the shard isolation boundary. Even for the polling gap, it's better to route through the processor than to give shards direct access to global lookup tables.

Fields stay nullable permanently for external/unknown addresses.
