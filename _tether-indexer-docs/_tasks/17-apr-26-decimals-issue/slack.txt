@Aliaksei Shaltykou @Eddy WM
From the ticket and code, the bug is a type/precision mismatch at the API /api/v2/notifications, not a template bug and not something only FE can fix.

We have 2 separate notification paths with different amount sources:

:white_check_mark: Successful path
"Transfer Successful" is emitted internally from the indexer (rumble-data-shard-wrk). The tx.amount arrives as a pre-formatted string from the EVM indexer, so interpolation is clean.

:x:  Failing path

"Token Transfer Initiated" is emitted by app POST to /api/v2/notifications. 
The schema at rumble-app-node here and here declares amount: { type: 'number' } which is a JS float64. 
The mobile app is computing rawUnits / 10 ** decimals in floating point (e.g. 26882800 / 1e9 === 0.026882800000000002), the backend schema accepts it, the ork layer (rumble-ork-wrk) here forwards it unchanged, and the template does ${amount} on the Number, exposing the IEEE-754 artifact verbatim.


Fix priority:
1. Mobile app: send pre-formatted amount as a string (formatUnits(raw, decimals)).
2. Schema: change amount to { type: 'string', pattern: '^[0-9]+(\\.[0-9]+)?$' } at both endpoints.
3. Template: defensive formatting with token-aware decimals. Don't hardcode toFixed(6) . This differs by chain/token.

This is not a trivial fix.
