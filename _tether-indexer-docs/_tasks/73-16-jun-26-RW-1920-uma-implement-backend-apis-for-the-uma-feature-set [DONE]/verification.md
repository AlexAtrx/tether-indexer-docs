## 0. Tether Wallet ALREADY has the full UMA feature set (`tether-wallet-app-node`)

Francesco's "the API matches what we have in tether wallet" is exactly right —
TW already ships every endpoint in this ticket. From
`tether-wallet-app-node/workers/lib/server.js` and `workers/lib/services/uma.js`
(copies in `tw-reference/`):

- `POST /api/v1/username/suggest` (`server.js:1002`, tag `UMA`, rate-limited
  `umaSuggest` 30/60s) → `service.uma.suggest`. Derives candidates from the
  email (`suggestCandidatesFromEmail`) and returns the first one not already
  taken (checked via `ork.lookupUmaUsername`). Body `{ email }` → `{ username }`.
- `POST /api/v1/username/check` (`server.js:1043`) → `service.uma.check`.
  Validates format then checks availability via `ork.lookupUmaUsername`. Returns
  `{ available, reason?, code? }`.
- `POST /api/v1/wallets` (`server.js` ~913-961) accepts an optional `username`
  per wallet, normalized/validated by `validateUsernameFormat` before
  `ork.addWallet`.
- `PATCH /api/v1/wallets/:id` (`server.js:964`) → `wdkService.ork.updateWallet`.
- `GET /api/v1/wallets` and `GET /api/v1/wallets/:id` return the wallet "with
  UMA (username, uma) when present" (`server.js:861`, `:888`).
- `GET /.well-known/lnurlp/:username` (`server.js:1088`) — keyed on **username**
  (not spark pubkey), full UMA lookup/pay: `minSendable`/`maxSendable`,
  `umaVersion: '1.0'`, `defaultSettlementLayer`, `settlementOptions`,
  identifier `$username@domain`. Plus `GET /api/lnurl/payreq/:uuid` and
  `POST /api/uma/payreq/:uuid`.
- Settlement layers: `lightning, ln, spark, ethereum, polygon, arbitrum, plasma,
  tron`; assets `btc, sat, usdt, xaut` (`utils/uma.js getUmaChainsConfig`).
- **Username rules (`utils/uma.js validateUsernameFormat`):** 4–15 chars,
  `^[a-z0-9]+$`, ≥1 digit, ≥1 letter, uppercase auto-lowercased. Error codes
  `ERR_USERNAME_{REQUIRED,LENGTH,INVALID_CHARS,MIN_DIGITS,MIN_LETTERS}`.
- **TW owns the username entirely.** `grep rumble` and `grep preferred_username`
  over `tether-wallet-app-node` = **zero hits**. suggest/check/storage all go
  through TW's own ork → data-shard (`lookupUmaUsername`, `getUmaByUsername`,
  `getUmaByUserId`). No Rumble call anywhere. This is precisely Alex's "in TW the
  username is handled by us."

**Implication for RW-1920:** the work is to bring this TW UMA surface to the
**Rumble** side (`rumble-app-node` currently keys lnurlp on the spark pubkey, has
no username endpoints), and to decide how the Rumble UMA username relates to
Rumble's SSO-owned `preferred_username` (see 1b). TW is the reference
implementation to port/share, not a greenfield design.

## 1. Alex's claim — CONFIRMED on the TW side; Rumble side still open

> "In TW the username is handled by us but on the Rumble side the usernames are
> generated by Rumble. Do we have to communicate with the Rumble APIs to sync
> the usernames in case the user sets or resets their usernames?"

What the code actually shows:

- **"In TW the username is handled by us" — TRUE.** `tether-wallet-app-node`
  owns the username end to end (suggest from email, format-validate, availability
  check, store, resolve for lnurlp) with no Rumble interaction at all — see
  section 0. So TW is the working reference for the wallet-side username
  lifecycle.
- **Where the username is stored:** TW persists it via its own ork → data-shard
  (`ork.lookupUmaUsername` / `getUmaByUsername` / `getUmaByUserId`, and
  `ork.addWallet` / `updateWallet` carry the `username`). The `uma` object on a
  wallet holds at least `{ username, domain, walletId, userId, minSendable,
  maxSendable, defaultSettlementLayer }`. (Note: the *local* `wdk-app-node` base
  and its `wdk-data-shard-wrk` wallet entity have no `username` field — UMA lives
  in the TW/Rumble product layers above the open base, which is why the local
  grep was empty.)
- **"On the Rumble side usernames are generated by Rumble" — TRUE.** Rumble's
  SSO issues the username as the OIDC `preferred_username` claim (see 1b). That
  is a *different* username namespace from TW's UMA handle.
- **A Rumble sync channel already exists** and is the natural home for any
  push-to-Rumble: `rumble-data-shard-wrk/workers/lib/utils/rumble.server.util.js`
  posts signed webhooks (`x-signature` / `x-signed-on`) to
  `${conf.rumbleServer.url}/-wallet/webhook/...` — today `transaction-init`,
  `transaction-complete`, `jar-sync`. There is **no** username-sync route yet on
  either side.

**Conclusion:** Alex's framing is correct — TW self-manages the UMA username,
Rumble generates the login `preferred_username`. The open question is purely on
the **Rumble integration**: should the Rumble UMA username be (A) derived from /
kept in sync with Rumble's `preferred_username`, or (B) a standalone TW-style
handle that, on set/reset, syncs back to Rumble? TW gives no precedent for
syncing (it never talks to Rumble), so this is a Rumble-side decision. **Still
needs confirmation from Alex / Eddy / Rumble** plus the Rumble endpoint contract
if a sync is required.

## 1b. Does username management involve Rumble APIs? — YES (Rumble owns the login username)

Checked the Rumble API specs (`rumble-docs/api/...` Bruno collection) and the
wallet-backend code that talks to Rumble. The evidence is decisive:

- **The login username is a Rumble identity, generated and owned by Rumble's
  SSO.** `rumble-app-node` proxies auth straight to Rumble's IdP via
  `service.rumble.proxyRumbleAuth` (`rumble-app-node/workers/lib/services/rumble.js`),
  which forwards to `ctx.conf.sso.authUrl` using a client-id/secret Basic header
  (`config/common.json.example` → `sso.{authUrl,clientId,clientSecret}`). The
  proxied surface is passkey + OAuth: `/passkey/registration/{start,finish}`,
  `/passkey/authentication/{start,start/with-email,finish}`,
  `/api/auth/authorization-code`, `/api/auth/refresh-token`
  (`rumble-app-node/workers/lib/server.js:425-507`).
- **The OIDC `id_token` Rumble returns carries the username.** Decoding the
  sample token in `rumble-docs/api/Proxy to Rumble/POST -api-auth-authorization-code.bru`:
  `iss: https://auth.rumble.com`, `sub: 264749945`,
  `preferred_username: codewithabhijit`, `email`, `name`, `scope: openid email
  profile wallet`. Passkey registration takes `{ username, password }` and the
  response echoes `user.name`/`displayName` (e.g. `ramzzz3s`). So Rumble's auth
  server is the source of truth for that username. This is exactly Alex's "on
  the Rumble side the usernames are generated by Rumble."
- **But the wallet backend does not read that username today.** `RumbleJwtGuard._parsePayload`
  (`rumble-app-node/workers/lib/middlewares/rumble.jwt.guard.js:6`) returns only
  `{ id: payload.userId || req.params/query/body.userId }`; the base guard
  (`wdk-app-node/.../auth/jwt.guard.js:62`) returns `{ id: payload.userId }`.
  Neither reads `sub` or `preferred_username`. Everything is keyed on `userId`;
  the username claim is currently ignored.
- `grep preferred_username` across all backend repos = **zero hits**. No code
  consumes, stores, checks, or syncs a username anywhere.

**So there are two distinct username concepts, and that is the heart of Alex's
question:**

1. The **Rumble login username** (`preferred_username`) — generated/owned by
   Rumble's SSO, already present in the id_token, not consumed by the wallet
   backend.
2. The **UMA username** RW-1920 wants — the localpart of `username@domain`,
   which the spec wants the wallet backend to `suggest` / `check` / `claim` /
   set/reset and store ("in TW the username is handled by us").

The design decision RW-1920 must settle:

- **(A) Reuse the Rumble username as the UMA username.** Then `/username/suggest`
  and `/username/check` and any set/reset must defer to Rumble's APIs (Rumble
  owns that namespace and already enforces uniqueness at registration). The
  wallet backend would read `preferred_username` from the id_token (cheap, no
  new Rumble call) and/or call a Rumble user API to mutate it. **No Rumble
  username-management endpoint is exposed in `rumble-docs` today**, so this path
  needs a new Rumble API.
- **(B) TW owns a separate UMA handle**, independent of the Rumble login name.
  Then TW owns suggest/check/claim/storage, and the only Rumble interaction is
  the optional sync Alex asked about — pushing set/reset to Rumble so the two
  stay aligned, via the existing signed-webhook channel
  (`rumble-data-shard-wrk/.../rumble.server.util.js`), which has no username
  route yet.

**Bottom line for the question asked:** yes, username management is tied to
Rumble APIs — Rumble is the identity provider and already owns/generates a
username (`preferred_username`) via its SSO. What is undecided is whether the
UMA username equals that Rumble username (→ must call Rumble to suggest/check/
set) or is a separate TW-owned handle that only needs to be synced back to
Rumble. Neither a Rumble username-management endpoint nor any wallet-side
username code exists today, so this must be agreed with the Rumble team before
implementation. This is the same blocker recorded in item 1.

## 3. PATCH /wallets — confirmed in both layers

- **In TW (`tether-wallet-app-node`):** `PATCH /api/v1/wallets/:id`
  (`server.js:964`), body `patchWalletsBodySchema`, → `wdkService.ork.updateWallet`.
  This is the endpoint Francesco referred to. TW already carries `username`
  through wallet create/update via `validateUsernameFormat` + `ork.addWallet` /
  `updateWallet`, so username set/reset on the wallet is a solved pattern there.
- **In the local WDK base (`wdk-app-node`):** `PATCH /api/v1/wallets/:id`
  (`server.js:558-599`), body strict (`additionalProperties: false`): `enabled?`,
  `name?`, `addresses?`, `accountIndex?`, `meta?` → `ork.updateWallet`
  (`services/ork.js:232-243`) → data-shard `updateWallet`
  (`wdk-data-shard-wrk/workers/proc.shard.data.wrk.js:399-529`). The **base** has
  no `username` (UMA is added in the TW layer above it).

Relation to the discussion: PATCH `/wallets/:id` is the home for username
set/reset (TW already does this). For the Rumble port, a set/reset here is the
event that would sync to Rumble **if** model (B) is chosen — via a new
`syncUsername` on `rumble-data-shard-wrk/.../rumble.server.util.js`, once the
Rumble-side endpoint is agreed. If model (A) is chosen (UMA username = Rumble
`preferred_username`), suggest/check/set must instead defer to a Rumble user API.

In the local `wdk-app-node` base the PATCH schema is `additionalProperties:
false`, so a `username` would be rejected there — but that base is not the TW
product. The TW product (`tether-wallet-app-node`) accepts and stores `username`.

- TW owns its UMA username with **no Rumble interaction**; Rumble separately owns
  the login `preferred_username` via SSO.
- **The one open blocker** is the Rumble integration decision (item 1 / 1b):
  is the Rumble UMA username (A) the same as Rumble's `preferred_username`
  (→ suggest/check/set must call a Rumble user API, which does not exist yet),
  or (B) a standalone TW-style handle that syncs back to Rumble on set/reset
  (→ new `syncUsername` webhook). Get that answer (and the UMA design doc)
  before coding.
