Hey Anton, good questions. Here are the answers:

1. Is getUserTransferHistory the replacement for getUserTokenTransfers?

Yes. getUserTransferHistory (GET /api/v1/users/:userId/transfer-history) is the replacement for getUserTokenTransfers (GET /api/v1/users/:userId/token-transfers). The legacy endpoint stays available for now, but all new work should use the new one. The new endpoint returns grouped, enriched transfers from a different collection (wallet_transfers_processed) with a richer response shape (swap classification, fiat amounts, fee labels, etc.).

2. When to use getUserTransferHistory vs getWalletTransferHistory?

- getUserTransferHistory: returns transfers across ALL wallets for a user. Use this for the main transaction history screen, the home screen feed, or anywhere you need a unified view of all the user's activity. It also supports a walletTypes query param if you want to filter by wallet type without specifying an exact wallet.

- getWalletTransferHistory: returns transfers for a SINGLE wallet. Use this when you're inside a specific wallet detail screen and want to show only that wallet's history.

Both endpoints share the same query params (token, blockchain, type, from, to, limit, skip, sort) and return the exact same response shape, so the mapping logic on your side is identical for both.

Regarding your proposed 2-phase plan — that makes sense. Phase 1 (swap getUserTokenTransfers for getUserTransferHistory, update the response mapping) gets you back to full functionality with the new data shape. Phase 2 (add getWalletTransferHistory where needed) can follow once wallet-specific screens are identified.
